# 系统架构师评审 SOP

> **版本**: v1.0  
> **生效日期**: 2026-04-18  
> **适用角色**: ADVISOR-SA (系统架构顾问)  
> **触发条件**: FEATURE_REQUEST / AUTOMATION_CREATE / AUTOMATION_OPTIMIZE

---

## 1. 概述

本 SOP 定义了 Dream-MultiSkill 系统中，所有 Skill、自动化任务、功能变更的标准化审查流程。确保技术安全、架构合规、知识沉淀可控。

### 1.1 评审目标

| 目标 | 说明 |
|:---|:---|
| **安全性** | 无恶意代码、数据泄露风险、权限滥用 |
| **架构合规** | 符合现有系统设计，不破坏模块边界 |
| **接口契约** | Skill 之间调用接口稳定、可追溯 |
| **知识沉淀** | 决策逻辑可审计，经验可复用 |

### 1.2 评审等级

| 等级 | 触发条件 | 评审深度 | 审批人 |
|:---:|:---|:---:|:---|
| **P0 - 紧急** | 安全漏洞、系统崩溃风险 | 5分钟快审 | ADVISOR-SA |
| **P1 - 高** | 新 Skill 安装、API 密钥配置 | 标准评审 | ADVISOR-SA + ADVISOR-RM |
| **P2 - 中** | 自动化任务创建/修改 | 简化评审 | ADVISOR-SA |
| **P3 - 低** | 文档更新、非代码变更 | 自审通过 | 提交者自检 |

## 2. 评审流程

### 2.1 标准评审流程 (P0/P1/P2)

```
┌─────────────────────────────────────────────────────────────────┐
│                        评审流程 (D4 工作日)                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [提交申请] → [ADVISOR-SA 初审] → [安全扫描] → [架构检查] → [审批]  │
│       ↓              ↓                ↓           ↓            │
│   提交者          24h内响应       自动执行      通过/拒绝         │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**Step 1: 提交申请**
- 填写评审申请表（见附录 A）
- 提供变更内容说明、影响范围、风险评估
- 附上相关代码/配置/文档

**Step 2: ADVISOR-SA 初审 (24h内)**
- 确认评审等级
- 分配评审资源
- 如为 P0，直接进入紧急通道

**Step 3: 安全扫描**
- 代码静态分析
- 敏感信息检测（API Key、密码、Token）
- 依赖包安全检查

**Step 4: 架构检查**
- 模块边界检查
- 接口契约验证
- 与现有 Skill 冲突检测

**Step 5: 审批决策**
- 通过 → 合并/发布
- 拒绝 → 附上修改建议
- 需修改 → 返回 Step 1

## 3. 各类评审标准

### 3.1 Skill 评审标准

| 检查项 | 要求 | 权重 |
|:---|:---|:---:|
| **命名规范** | 符合 `domain-purpose` 格式 | 10% |
| **触发词覆盖** | description 覆盖主要使用场景 | 15% |
| **权限最小化** | 只申请必要的能力权限 | 20% |
| **输出可追溯** | 有明确的 context_pack 输出格式 | 15% |
| **错误处理** | 有 fallback 机制和错误码 | 15% |
| **文档完整** | SKILL.md 包含所有必要章节 | 15% |
| **测试覆盖** | 有可运行的测试用例 | 10% |

**通过标准**: 综合得分 ≥ 70% 且无 P0 安全问题

### 3.2 自动化任务评审标准

| 检查项 | 要求 | 权重 |
|:---|:---|:---:|
| **触发条件明确** | rrule 或 scheduledAt 定义清晰 | 15% |
| **幂等性** | 重复执行不会产生副作用 | 25% |
| **错误恢复** | 失败后有降级或告警机制 | 20% |
| **资源限制** | 有 maxDurationMinutes 限制 | 15% |
| **日志可查** | 执行结果可追溯 | 15% |
| **业务价值** | 有明确的价值主张 | 10% |

**通过标准**: 综合得分 ≥ 65% 且无 P0 风险

### 3.3 功能变更评审标准

| 检查项 | 要求 | 权重 |
|:---|:---|:---:|
| **向后兼容** | 不破坏现有接口契约 | 30% |
| **版本管理** | 有清晰的版本号变更 | 10% |
| **文档同步** | 文档与代码同步更新 | 20% |
| **影响分析** | 明确影响范围和回滚方案 | 25% |
| **测试验证** | 有对应的测试用例 | 15% |

**通过标准**: 综合得分 ≥ 75% 且有回滚方案

## 4. 安全红线 (P0 项 - 一票否决)

以下情况 **直接拒绝**，不进入评审：

| 红线 | 说明 |
|:---|:---|
| 🔴 **硬编码密钥** | 代码中直接写入 API Key、密码、Token |
| 🔴 **删除围栏** | 移除 `围栏注入` 相关安全机制 |
| 🔴 **绕过审批** | 未经评审直接合并到主分支 |
| 🔴 **数据泄露** | 将用户数据发送到外部服务器 |
| 🔴 **权限提升** | 申请超出必要范围的系统权限 |

## 5. 接口契约规范

### 5.1 Skill 接口契约

每个 Skill 必须遵循以下契约：

```yaml
skill_interface:
  input:
    - context: "标准化的 context_pack"
    - parameters: "明确的参数定义"
  output:
    - result: "结构化的输出"
    - evidence_refs: "可追溯的证据引用"
    - confidence: "置信度评分"
  error:
    - error_code: "标准错误码"
    - fallback: "降级策略"
```

### 5.2 必需的头部注释

所有 Skill 脚本必须包含：

```python
"""
{Skill名称}
版本: {version}
作者: ADVISOR-{CODE}
功能: {一句话描述}
接口契约: {遵循的契约版本}
最后更新: {YYYY-MM-DD}
"""
```

## 8. 评审检查清单

### 8.1 Skill 评审检查清单

```yaml
skill_review_checklist:
  
  P0 检查 (5分钟):
    □ 无硬编码密钥
    □ 无数据外泄
    □ 无权限提升
    □ 权限最小化

  P1 检查 (30分钟):
    □ SKILL.md 格式规范
    □ description 触发词覆盖
    □ input/output 契约明确
    □ 错误处理完整
    □ 测试用例存在

  P2 检查 (15分钟):
    □ 符合命名规范
    □ 无明显漏洞
    □ 文档齐全
```

### 8.2 自动化评审检查清单

```yaml
automation_review_checklist:
  
  触发检查:
    □ rrule/scheduledAt 格式正确
    □ 执行频率合理
    □ 时区正确 (UTC+8)

  安全检查:
    □ 无外部副作用
    □ 日志可追溯
    □ 失败有告警
    □ 有超时限制

  业务检查:
    □ 幂等性验证
    □ 并发安全
    □ 资源消耗可控
```

### 8.3 功能变更评审检查清单

```yaml
feature_review_checklist:
  
  兼容性检查:
    □ 接口向后兼容
    □ 参数可选
    □ 错误码稳定

  影响分析:
    □ 影响范围明确
    □ 回滚方案存在
    □ 测试覆盖

  文档检查:
    □ CHANGELOG 更新
    □ API 文档同步
    □ 示例代码更新
```

## 9. 常见问题与处理

### 9.1 评审拒绝原因 Top 5

| # | 原因 | 解决方案 |
|:---:|:---|:---|
| 1 | 触发词描述模糊 | 补充具体使用场景 |
| 2 | 缺少错误处理 | 添加 try-except 和 fallback |
| 3 | 无测试用例 | 编写基础测试 |
| 4 | 权限过宽 | 最小化权限请求 |
| 5 | 接口不兼容 | 使用标准化契约 |

### 9.2 快速修复指南

```yaml
常见问题 → 修复方案:

  Q: description 太笼统
  A: 添加具体触发词和使用场景

  Q: 缺少错误码
  A: 定义标准化错误码表

  Q: 无 fallback
  A: 添加降级到默认策略

  Q: 权限申请过多
  A: 拆分 Skill 或最小化权限

  Q: 无日志
  A: 添加结构化日志输出
```
