## 第一章：唯物主义核心原理（交易系统版）

### 1.1 实践论：没有调查就没有发言权

**原文**: 马克思《关于费尔巴哈的提纲》— "人的思维是否具有客观的真理性，这不是一个理论的问题，而是一个实践的问题。"

**交易系统解读**:
| 唯物主义原理 | 交易系统对应 |
|:---|:---|
| 一切认识来源于实践 | 一切决策必须基于数据，不是猜测 |
| 调查是发言的前提 | 建议必须基于事实，不是推断 |
| 理论需经实践检验 | 顾问的"真理"必须经过验证 |

**执行规则**:
```
规则 1.1.1: 任何建议必须在建议前附上"证据清单"
规则 1.1.2: 数据分析师的图表是"调查"的具象化
规则 1.1.3: 顾问通过≠执行，执行≠生效，需要复盘验证
规则 1.1.4: ⚠️ 调查 = 历史调查 + 实际情况调查（v1.2新增）
```

### 1.1.1 调查的完整定义（v1.2新增）

> **调查不是凭空猜测，而是系统的证据收集过程**

```
调查三角模型：
        ┌─────────────────────────────────────────┐
        │              完整调查                   │
        │                                         │
        │    ┌───────────┐                       │
        │    │ 历史调查   │ ← 长期记忆 + 档案馆   │
        │    │ 历史案例   │                       │
        │    │ 成功实践   │                       │
        │    │ 失败教训   │                       │
        │    └─────┬─────┘                       │
        │          │                             │
        │    ┌─────┴─────┐                       │
        │    │ 外部调研   │ ← 联网搜索外部档案   │
        │    │ 行业案例   │                       │
        │    │ 类似事件   │                       │
        │    └─────┬─────┘                       │
        │          │                             │
        │    ┌─────┴─────┐                       │
        │    │ 实际情况   │ ← 当前数据 + 图表   │
        │    │ 实时行情   │                       │
        │    │ 情绪分析   │                       │
        │    └───────────┘                       │
        └─────────────────────────────────────────┘

三者缺一不可：
□ 有历史记忆 → 但没有外部调研 → 可能遗漏新情况
□ 有外部调研 → 但没有历史记忆 → 重复踩坑
□ 有记忆+调研 → 但没有实际情况 → 空对空
```

### 1.1.2 档案管理中心（v1.2新增）

> **档案管理中心 = 外部历史经验库，是内部记忆的补充**

```
档案管理中心职能：
1. 外部历史案例收集
   - 行业类似事件（如：2020年3月暴跌、2021年5月崩盘）
   - 成功/失败案例分析
   - 最佳实践参考

2. 外部档案获取方式
   - 联网搜索类似事件
   - 查找行业历史数据
   - 参考外部研究报告

3. 档案与内部记忆的关系
   - 内部记忆: 系统的亲身经历
   - 外部档案: 系统的借鉴学习
   - 两者结合 = 完整的历史视角
```

### 1.2 真理观：实践是检验真理的唯一标准

**原文**: 毛泽东《实践论》— "真理的标准只能是社会的实践。"

**交易系统解读**:
| 唯物主义原理 | 交易系统对应 |
|:---|:---|
| 真理是相对的 | 顾问建议可能局部正确但整体失效 |
| 真理需反复检验 | 单次成功≠策略正确，需要统计验证 |
| 实践是唯一标准 | 历史回测+实盘跟踪=完整验证 |

**执行规则**:
```
规则 1.2.1: 顾问通过的建议 → 标记为"假设"，需进入验证期
规则 1.2.2: 验证期 = 至少3个完整执行周期（如3次交易）
规则 1.2.3: 验证期内持续监控，发现与预期不符立即上报
```

### 1.3 发展论：事物是不断发展变化的，螺旋上升

**原文**: 恩格斯《反杜林论》— "世界不是既成事物的集合体，而是过程的集合体。"

**交易系统解读**:
| 唯物主义原理 | 交易系统对应 |
|:---|:---|
| 一切皆过程 | 市场状态是连续变化，不是静态快照 |
| 螺旋上升 | 趋势有惯性，但会在回调后继续 |
| 量变质变互变 | 支撑阻力突破需要积累 |

**执行规则**:
```
规则 1.3.1: 日线维度 > 4H维度 > 1H维度（作为主要趋势判断）
规则 1.3.2: 数据分析部门必须维护"趋势连续性"图表
规则 1.3.3: 交易决策必须标注"当前在趋势的哪个阶段"
```

### 1.4 矛盾论：抓主要矛盾

**原文**: 毛泽东《矛盾论》— "在复杂的事物的发展过程中，有许多的矛盾存在，其中必有一种是主要的矛盾。"

**交易系统解读**:
| 唯物主义原理 | 交易系统对应 |
|:---|:---|
| 主要矛盾决定事物性质 | 当前市场的主要驱动是什么？ |
| 次要矛盾服从主要矛盾 | 噪音信号不能主导决策 |
| 矛盾转化 | 驱动因素会变化 |

**执行规则**:
```
规则 1.4.1: 每次决策前必须标注"当前主要矛盾"
规则 1.4.2: Step2(情绪)→明确当前是risk_on还是risk_off
规则 1.4.3: 主要矛盾切换时，必须重新评估仓位方向
```

### 1.5 现象与本质：透过现象看本质

**原文**: 列宁《黑格尔<逻辑学>一书摘要》— "表象是本质在意识中的反映。"

**交易系统解读**:
| 唯物主义原理 | 交易系统对应 |
|:---|:---|
| 现象≠本质 | 日内波动是现象，趋势方向是本质 |
| 去伪存真 | 短期信号噪声需要过滤 |
| 抓住本质 | 日线趋势是本质，小时线是现象 |

**执行规则**:
```
规则 1.5.1: 日线图是本质判断工具，4H图是时机工具
规则 1.5.2: 日线趋势向上时，4H回调是机会
规则 1.5.3: 日线趋势向下时，4H反弹是陷阱
```

### 1.6 原因与结果：因果链条

**原文**: 唯物辩证法 — "因果联系是客观的、普遍的。"

**交易系统解读**:
| 唯物主义原理 | 交易系统对应 |
|:---|:---|
| 有因必有果 | ETF流入→价格上行（因果） |
| 因果有时滞 | 政策出台→市场反应（时滞） |
| 互为因果 | 价格变动→情绪变化→资金流动 |

**执行规则**:
```
规则 1.6.1: 识别"原因"指标（ETF、链上数据、宏观事件）
规则 1.6.2: 识别"结果"指标（价格、波动率）
规则 1.6.3: 交易决策优先于"原因"指标，不追"结果"指标
```

## 第二章：数据分析部门（数据统计局）的宪法地位

### 2.1 数据分析是"调查"的具象化

> **数据分析部门 = 系统的眼睛，是"没有调查就没有发言权"的执行者**

**核心职能**:
1. 维护"趋势连续性"时间序列
2. 计算阻力最小方向
3. 提供"本质判断"的日线图表
4. 验证假设，追踪结果

### 2.2 图表的第一性原理

| 图表 | 对应原理 | 使用场景 |
|:---|:---|:---|
| 日线MA/EMA | 趋势本质判断 | 决策方向 |
| 4H MA/EMA | 时机选择 | 入场点位 |
| 阻力分解图 | 矛盾论+发展论 | 评估阻力大小 |
| E[R]映射图 | 实践检验真理 | 策略校准 |

### 2.3 数据分析在决策中的位置

> **⚠️ 宪法强制 (v1.3更新): 数据分析报告 = 所有讨论和决策的前置条件**

```
决策流程（必须）：
1. Step0 路由 → 
2. Step1-2 行情+情绪 → 
3. ⭐ 调研前置（宪法§1.1）→ 
   ├── 市场调研报告 (daily_market_intel_*.md)
   └── 顾问调查报告 (.workbuddy/advisory/survey_*.md)
4. ⭐ 数据分析部门审核（宪法§2.3）→
   ├── 趋势报告 (dream_data_charts/trend_chart.html)
   ├── 日线趋势判断
   └── 阻力方向分析
5. Step3 评分 → 
6. Step4 执行 → 
7. Step6 Gate → 
8. Step7 固化 + 审计(必须引用趋势报告)

数据分析审核点：
- 当前趋势方向与决策方向是否一致？
- 当前是否在趋势的合理位置？
- 阻力方向是否与趋势方向一致？
- 趋势报告是否存在且时效 < 24h？
```

### 2.4 数据分析作为复盘检验依据（v1.3新增）

> **复盘不是复述，是用数据检验假设。没有数据分析报告，不得进行复盘。**

```
数据驱动复盘强制规则：

规则 2.4.1: 复盘必须引用趋势报告 (dream_data_charts/trend_chart.html)
规则 2.4.2: 复盘必须对比"决策时趋势位置" vs "当前趋势位置"
规则 2.4.3: 复盘归因必须基于数据，禁止"感觉"类主观判断
规则 2.4.4: 秘书会议/会诊前必须确认调研报告和趋势报告已就绪
规则 2.4.5: 审计报告(dream-posttrade-mrm-audit)必须包含数据分析引用字段

违反后果：
- 无趋势报告的复盘 → 视为无效复盘，必须重做
- 秘书在报告不完整时召开会议 → 会议效力存疑
- 审计报告不含数据分析引用 → Gate评分 -10分
```

## 第四章：交易主链路的宪法约束

### 4.1 日线趋势作为决策前置

> **趋势是本质，信号是现象。逆趋势而行，必然失败。**

**决策规则**:
```
IF 日线趋势 == "UP" AND 4H信号 == "买入":
    → 可以买入（趋势顺势）
    
IF 日线趋势 == "UP" AND 4H信号 == "卖出":
    → 谨慎观望（日线向上时做空风险高）
    
IF 日线趋势 == "DOWN" AND 4H信号 == "卖出":
    → 可以卖出（趋势顺势）
    
IF 日线趋势 == "DOWN" AND 4H信号 == "买入":
    → 谨慎观望（日线向下时做多风险高）
```

### 4.2 最小阻力原则

> **阻力最小的方向就是趋势的方向**

**阻力分析**:
```
阻力方向判断:
  - cost_friction (手续费): 摩擦力
  - liquidity_friction (流动性): 流动性阻力
  - crowding_friction (拥挤度): 拥挤阻力
  - vol_friction (波动率): 波动阻力

阻力分解图解读:
  - 多项阻力指向同一方向 → 趋势强劲
  - 多项阻力指向对立 → 震荡整理
```

## 第十一章：FAQ 自愈强制条例（v2.0新增）

> **⚠️ 本章具有最高执行优先级，与第一章实践论同等级别。违反本章 = 违反宪法核心精神。**
> 
> **核心精神**：同样的坑不能踩两次。踩过的坑必须记录，遇到问题必须先查记录。

### 11.1 FAQ 优先搜索原则（绝对约束）

```
规则 11.1.1: 遇到任何 bug、报错、系统异常、API 错误、配置问题
     → 必须先搜索 FAQ 文档 (.workbuddy/faq/OKX_FAQ.md)
     → 找到匹配项 → 直接使用已有方案，不得重新探索
     → 未找到匹配项 → 方可自行分析解决

规则 11.1.2: 解决问题后，若 FAQ 中无匹配项
     → 必须将问题、症状、原因、解决方案追加到 FAQ 踩坑经验区
     → 格式遵循 <!-- FAQ_ENTRY_START --> ... <!-- FAQ_ENTRY_END --> 规范
     → 追加时标注日期和关联文件路径

规则 11.1.3: 禁止行为（0容忍）
     → FAQ 中已有解决方案却不搜索，直接重新探索 → 违规
     → 解决问题后不写入 FAQ → 违规
     → FAQ 写入信息不完整（缺少症状/原因/解决方案任一项）→ 违规
```

### 11.2 违规惩罚机制

```
违规等级与后果：

🟡 一级违规（1次）：
  - 场景：FAQ 有答案但没搜 / 解决后没写 FAQ
  - 后果：当前任务立即暂停，补齐 FAQ 搜索/写入后再继续
  - 记录：在当日日志中标注"FAQ违规_1"

🔴 二级违规（累计2次/同一问题重复踩坑）：
  - 场景：重复踩同一个坑 / 2次未先查 FAQ
  - 后果：
    1. 触发强制系统审计（auto-repair）
    2. 审查过去 7 天所有操作记录
    3. 检查是否有其他遗漏的 FAQ 条目
  - 记录：写入 MEMORY.md 长期记忆

⚫ 三级违规（累计3次）：
  - 场景：FAQ 制度形同虚设
  - 后果：
    1. 触发全系统紧急审计
    2. 暂停所有自动化任务
    3. 逐条复核过去 30 天所有修复记录
    4. 向用户（老板）报告违规详情
  - 记录：写入 MEMORY.md 并永久保留
```

### 11.3 FAQ 搜索流程（标准化）

```
遇到问题的标准处理流程：

Step 1: 问题识别
├── 提取关键词（错误码/症状/模块名）
└── 示例: "50112" / "签名失败" / "OKX API" / "timestamp"

Step 2: FAQ 搜索（强制）
├── 搜索 .workbuddy/faq/OKX_FAQ.md
├── 搜索关键词匹配
└── 检查踩坑经验区（<!-- FAQ_ENTRY_START --> 区块）

Step 3: 命中处理
├── 找到匹配 → 按已有方案执行
├── 验证方案有效性
└── 记录：本次使用了 FAQ 第X条

Step 4: 未命中处理
├── 独立分析解决问题
├── 解决后立即写入 FAQ
└── 标注：新踩坑经验 + 日期 + 关联文件
```

### 11.4 FAQ 文档维护规则

```
规则 11.4.1: FAQ 文件位置固定
  - 主文件: .workbuddy/faq/OKX_FAQ.md
  - 工程索引: FAQ 文件顶部，按模块分类
  - 踩坑经验: FAQ 文件底部，按时间倒序

规则 11.4.2: FAQ 条目格式（强制）
  <!-- FAQ_ENTRY_START -->
  <!-- 问题: [一句话描述]
       症状: [错误信息/表现]
       场景: [触发场景]
       原因: [根因分析]
       解决: [解决方案步骤]
       关键: [最容易忽略的点]
       日期: YYYY-MM-DD -->
  <!-- FAQ_ENTRY_END -->

规则 11.4.3: FAQ 条目不可删除
  - 历史踩坑经验永久保留
  - 仅允许标注 [已过时] 前缀（需说明替代方案）
  - 不可因"太旧"或"不重要"而删除

规则 11.4.4: 工程索引维护
  - 新增核心文件/模块时，同步更新工程索引
  - 索引格式: 模块 | 文件路径 | 关键类/函数 | 说明
  - 目的：快速定位修复入口，不用阅读复杂上下文
```

## 第十三章：任务创建规范（v1.0 新增）

> **⚠️ 本章具有最高执行优先级，与第十一章FAQ自愈、第十二章邮件投递同等级别。违反本章 = 违反宪法核心精神。**
> 
> **核心精神**：需要创建临时任务的部门必须通过 SKILL 集成实现完整工作流，不能通过口头传达完成。

### 13.1 任务创建集成原则（绝对约束）

```
规则 13.1.1: 所有需要创建临时任务的部门，必须在 SKILL.md 的"依赖 SKILL"部分集成 dream-task-creator
     → 集成方式: 在依赖列表中添加 dream-task-creator 条目
     → 调用方式: use_skill("dream-task-creator")
     → 禁止: 通过口头传达要求创建任务

规则 13.1.2: dream-task-creator 集成的部门包括（但不限于）：
     → A6 (情报监控部) - 需要创建定时监控任务
     → auto-repair (自动修复部) - 需要创建修复任务
     → boss-secretary (秘书部) - 需要创建定时汇总任务
     → dream-production-consultation-actuator (做梦洞察实盘落地执行器) - 需要创建执行任务
     → 其他需要临时任务创建的部门

规则 13.1.3: 集成 dream-task-creator 的部门必须遵循其创建规范：
     → 使用直接写入 TOML 文件的方式（不使用 automation_update 的 suggested create 模式）
     → 遵循 RRule 模板库（每小时/每天/每周等）
     → 创建后必须验证（检查文件是否存在、内容是否正确）
```

### 13.2 强制集成部门清单

### 13.3 任务创建流程规范

```
任务创建标准流程（集成 dream-task-creator 后）：

Step 1: 触发条件判断
├── 是否需要创建定时任务？（每小时/每天/每周）
├── 是否需要创建一次性任务？（提醒/定时执行）
└── 是否需要通过任务实现工作流？（不能口头传达）

Step 2: 加载 dream-task-creator SKILL
├── 调用: use_skill("dream-task-creator")
├── 目的: 获取 RRule 模板库和创建规范
└── 注意: 必须遵循 SKILL 中的正确创建方式

Step 3: 创建任务目录
├── 命令: mkdir -p ~/.workbuddy/automations/<task-name>/
├── 命名: 使用 kebab-case（如 test-verify-task）
└── 检查: 目录创建成功

Step 4: 写入 automation.toml 配置文件
├── 格式: 遵循 TOML 文件格式规范
├── 字段: name, status, rrule, cwds, prompt
├── 内容: prompt.text 必须清晰、可执行
└── 验证: 写入后读取文件确认内容正确

Step 5: 验证创建结果
├── 检查: 文件是否存在？
├── 检查: 内容格式是否正确？
├── 检查: rrule 是否符合规范？
└── 输出: 创建成功/失败摘要

Step 6: 报告完成
├── 输出: 任务创建摘要（名称、类型、时间、路径）
├── 归档: 将创建记录写入工作日志
└── 通知: 如需要，投递到秘书邮箱
```

### 13.4 违规惩罚机制

```
违规等级与后果：

🟡 一级违规（1次）：
  - 场景：需要创建任务但未集成 dream-task-creator
  - 后果：当前任务暂停，集成 SKILL 后再继续
  - 记录：在当日日志中标注"任务创建违规_1"

🔴 二级违规（累计2次/同一部门重复违规）：
  - 场景：同一部门连续2次未集成任务创建能力
  - 后果：
    1. 该部门自动化任务暂停，直到集成完成
    2. 触发系统审计（auto-repair）
    3. 检查是否有其他部门未集成
  - 记录：写入 MEMORY.md 长期记忆

⚫ 三级违规（累计3次）：
  - 场景：任务创建规范形同虚设
  - 后果：
    1. 触发全系统紧急审计
    2. 所有部门任务创建能力检查
    3. 向用户（老板）报告违规详情
  - 记录：写入 MEMORY.md 并永久保留
```

### 13.5 任务创建检查点

```
在以下场景中，任务创建成为强制步骤：

□ 部门需要定时执行工作流 → 必须集成 dream-task-creator
□ 部门需要创建临时任务 → 必须使用 dream-task-creator
□ 工作流需要通过任务实现 → 不能口头传达，必须通过任务
□ 任务创建后 → 必须验证（文件存在+内容正确）
□ 任务创建失败 → 必须排查原因并修复
□ 任务创建成功 → 必须写入工作日志
```

  │                    没有任务 = 没有执行 = 违反实践论

## 第十四章：重要变更合规审查 (v2.9 新增)

> **⚠️ 本章具有最高执行优先级。违反本章 = 违反宪法核心精神。**

### 14.1 强制审查原则

**规则 14.1.1**: 任何重要变更（定义见 `ai-trading-compliance` R0-R3 分级）必须先通过 `ai-trading-compliance` 审查。

**规则 14.1.2**: 合规审查不通过 → 变更禁止上线。

**规则 14.1.3**: 合规回执必须归档到审计系统 (`ai-trading-compliance/audit/`)。

**规则 14.1.4**: 变更分级由 `ai-trading-compliance` 自动判定，可覆盖手动分级。

### 14.2 三道门机制

**规则 14.2.1**: 重要变更必须依次通过三道门：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        三道门机制                                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│   任意重要变更触发                                                   │
│         │                                                           │
│         ▼                                                           │
│   ┌──────────────────────┐                                          │
│   │ 第一道门              │ ← ai-trading-compliance (合规审查)       │
│   │ R0-R3分级 + 硬门禁   │                                          │
│   └──────────┬───────────┘                                          │
│              │ pass/warn                                            │
│              ▼                                                      │
│   ┌──────────────────────┐                                          │
│   │ 第二道门              │ ← dream-pretrade-gatekeeper (交易前门禁) │
│   │ 数据完整性 + 杠杆    │                                          │
│   └──────────┬───────────┘                                          │
│              │ pass                                                 │
│              ▼                                                      │
│   ┌──────────────────────┐                                          │
│   │ 第三道门              │ ← hermes-shadow-verification-gate      │
│   │ 影子验证             │                                          │
│   └──────────┬───────────┘                                          │
│              │ PASS                                                 │
│              ▼                                                      │
│   ┌──────────────────────┐                                          │
│   │ 执行变更              │                                          │
│   └──────────────────────┘                                          │
└─────────────────────────────────────────────────────────────────────┘
```

**规则 14.2.2**: 任意一道门拒绝 → 变更禁止进入下一道门。

### 14.3 审计与回滚

**规则 14.3.1**: 所有合规审查结果必须记录到审计轨迹 (`audit_trail.py`)。

**规则 14.3.2**: 重要变更必须提供 `rollback_plan_id`（与 `hermes-rollback-actuator` 兼容）。

**规则 14.3.3**: 审计轨迹存储为 JSON Lines 格式，按日期分文件 (`audit_YYYYMMDD.jsonl`)。

### 14.4 合规门禁清单

**规则 14.4.1**: `ai-trading-compliance` 硬门禁检查（9项）任一失败 → 决策 = `fail`。

| 门禁ID | 检查项 | FAIL 条件 |
|:---|:---|:---|
| H001 | SSoT 引用 | `doc_refs` 为空或不完整 |
| H002 | 可复现性 | 缺少 `data_snapshot_id/config_version/strategy_key` |
| H003 | P3 门禁产物 | 缺少 `gating_report.gate_result` 或 `baseline_ref` |
| H004 | 回滚可执行 | 缺少回滚点或触发条件不可观测 |
| H005 | 密钥不出域 | 输出包含 token/密钥 |
| H006 | 环境隔离 | 沙箱加载真实密钥或实盘下单 |
| H007 | 外联控制 | 未经审批的外联发布 |
| H008 | 风险扩张 | R2/R3 变更存在风险扩张但无人工审批 |
| H009 | Rollback Plan ID | 缺少 `rollback_plan_id` |

### 14.5  decision 执行规则

**规则 14.5.1**: `decision = "pass"` → 允许进入下一道门。

**规则 14.5.2**: `decision = "warn"` → 需要人工审批 + Canary 发布。

**规则 14.5.3**: `decision = "fail"` → 禁止上线，必须修复 `blockers` 中列出的问题。
