## 二、违规类型与判定

### 2.1 违规类型定义

| 违规编号 | 违规类型 | 定义 | 严重度 |
|:---|:---|:---|:---:|
| **V001** | 跳步违规 | 未按Step 1-4处理直接跳到Step 4 | 🔴 |
| **V002** | FAQ未查阅 | 问题存在但未查FAQ | 🔴 |
| **V003** | FAQ未更新 | FAQ无解但未在48h内补充 | 🟠 |
| **V004** | 归档缺失 | 问题解决但未归档到FAQ/治理文档 | 🟡 |
| **V005** | 重复违规 | 同类问题累计3次跳步 | 🔴 |
| **V006** | 伪造记录 | 虚报已查阅FAQ | 🔴 |

### 2.2 违规判定规则

```python
# 违规判定伪代码
def check_violation(skill_name, problem_type, steps_taken):
    """
    判断是否违规
    """

    # V001: 跳步违规检测
    if steps_taken[0] != 'faq_check':
        if problem_type in EXISTING_FAQ_SECTIONS:
            return 'V001', '跳步违规 - 未查FAQ'

    # V002: FAQ未查阅
    if not has_checked_faq(problem_type):
        return 'V002', 'FAQ未查阅'

    # V003: FAQ未更新
    if FAQ_NO_ANSWER and not FAQ_UPDATED_48H:
        return 'V003', 'FAQ未更新'

    # V004: 归档缺失
    if PROBLEM_SOLVED and not ARCHIVED:
        return 'V004', '归档缺失'

    # V005: 重复违规
    if same_problem_violated_3_times():
        return 'V005', '重复违规'

    return None, None
```

## 三、处罚执行标准

### 3.1 处罚矩阵

| 违规编号 | 首次 | 二次 | 三次 | 累计五次 |
|:---|:---|:---|:---|:---|
| **V001** | 警告 | 记过+复盘 | 绩效-20% | 系统审计 |
| **V002** | 警告 | 记过 | 绩效-10% | 系统审计 |
| **V003** | 通知 | 警告 | 记过 | 审计 |
| **V004** | 记录 | 通知 | 警告 | 审计 |
| **V005** | 绩效-30% | 强制培训 | 降级 | 解聘 |

### 3.2 处罚执行流程

```
违规判定
    ↓
记录到 compliance_audit/penalty_records.md
    ↓
执行处罚（警告/记过/绩效扣减）
    ↓
通知SKILL负责人
    ↓
要求整改（24-72h内）
    ↓
整改验收
    ↓
关闭记录
```

每个SKILL的SKILL.md必须包含以下章节：

```markdown

## 【合规要求】（强制章节）

### §X 问题处理流程

> ⚠️ **合规约束**: 遇到任何问题必须按以下顺序处理：

```
遇到问题
    ↓
Step 1️⃣ 查FAQ
  → WORKSPACE/.workbuddy/faq/OKX_FAQ.md（OKX相关）
  → WORKSPACE/.workbuddy/faq/技术_FAQ.md（技术相关）
  → WORKSPACE/.workbuddy/faq/运营_FAQ.md（运营相关）

    ↓ 无解 → Step 2

Step 2️⃣ 查治理文档
  → ~/.workbuddy/skills/dream-governance-manager/governance_docs/
    ↓ 有解 → 执行 + 补充FAQ ✓
    ↓ 无解 → Step 3

Step 3️⃣ 联网搜索
  → 使用 agent-reach / tavily 搜索
    ↓ 有解 → 执行 + 归档经验 ✓
    ↓ 无解 → Step 4

Step 4️⃣ 自主分析
    ↓ 有解 → 执行 + 输出报告 + 归档 ✓
    ↓ 无解 → 升级处理
