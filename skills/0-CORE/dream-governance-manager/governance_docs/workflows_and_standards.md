## 一、决策流程规范

### 1.1 双轨决策路径（宪法第二章·补充）

```
┌─────────────────────────────────────────────────────────┐
│               双轨决策处理路径                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Problem Detected                                        │
│       │                                                 │
│       ├── P0/P1 交易类 ───→ A1→A2→A3→A4→A5            │
│       │                     │                             │
│       │                     ├─ 立即微信推送                │
│       │                     ├─ 等待30min                  │
│       │                     └─ 无回复 → 自主执行          │
│       │                                                 │
│       └── P2/常规类 ───→ 调研→顾问会诊→提案→执行      │
│                                │                       │
│                                └─ 顾问≥2/3同意        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 A1-A5 交易主链路规范

| 步骤 | 部门 | 核心产出 | 前置条件 |
|------|------|----------|----------|
| A1 | 情报监控部 | 市场情报报告 | - |
| A2 | 战略制定部 | 第一性原理分析 | A1完成 |
| A3 | 战术验证部 | 多情景推演方案 | A2完成 + 数据分析报告 |
| A4 | 战术验证部 | 验证报告 + 委托方案 | A3完成 + 三层索引 |
| A5 | 战术执行部 | 执行结果 | A4完成 + A7门禁通过 |

**强制前置条件（宪法§2.3）**:
- 数据分析报告（trend_chart.html）必须存在且时效 < 24h
- 市场调研报告（daily_market_intel_*.md）必须存在
- 顾问调查报告（survey_*.md）时效满足要求

## 二、信息流与投递规范

### 2.1 三邮箱体系（宪法§12.2）

```
┌─────────────────────────────────────────────────────────┐
│                  三邮箱体系                             │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ① 秘书邮箱 (secretary)                                 │
│     路径: ~/.workbuddy/skills/boss-secretary/reports/   │
│     用途: 收集各部门工作总结（唯一输入源）                │
│     消费: Secretary-Processor (每2h)                    │
│                                                         │
│  ② 调研部邮箱 (research)                                │
│     路径: ~/.workbuddy/skills/boss-secretary/reports/   │
│          research/                                      │
│     用途: 接收P1/P2/P3问题，等待调研                   │
│     消费: Research-Processor (每8h)                    │
│                                                         │
│  ③ 待修复邮箱 (pending)                                 │
│     路径: ~/.workbuddy/skills/boss-secretary/           │
│          pending_tasks/inbox/                           │
│     用途: 接收修复建议和待处理任务                       │
│     消费: Auto-Repair-Processor (每8h)                 │
│                                                         │
│  P0交易不走邮箱，直通A1-A5专属通道                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.2 投递规范（宪法§12.1）

```
规则 W.1: 所有部门完成工作后必须投递到秘书邮箱
规则 W.2: 文件命名: {部门标识}_{主题}_{YYYYMMDD_HHMMSS}.md
规则 W.3: 正文必须包含: 优先级标记 + 摘要 + 正文 + 建议
规则 W.4: P0交易完成后直接写入执行记录，不走邮箱
规则 W.5: 投递是工作完成的最后一步，未投递 = 工作未完成
```

## 三、FAQ 与问题处理规范

### 3.1 问题处理优先级（宪法§11.1）

```
遇到问题 → Step1: 查FAQ
         ↓ 有解 → 直接执行 ✓
         ↓ 无解 → Step2: 查治理文档
                   ↓ 有解 → 执行 + 补充FAQ ✓
                   ↓ 无解 → Step3: 联网搜索
                             ↓ 有解 → 执行 + 归档经验 ✓
                             ↓ 无解 → Step4: 自主分析
                                       ↓ 有解 → 执行 + 输出报告 + 归档 ✓
                                       ↓ 无解 → 升级处理
```

### 3.2 FAQ 维护规范（宪法§11.4）

| 规则 | 内容 |
|------|------|
| 文件位置 | `.workbuddy/faq/OKX_FAQ.md`（主文件） |
| 条目格式 | `<!-- FAQ_ENTRY_START --> ... <!-- FAQ_ENTRY_END -->` |
| 更新触发 | 任何问题被解决后，若FAQ中无匹配项 |
| 删除规则 | FAQ条目不可删除，仅允许标注`[已过时]` |
| 索引维护 | 新增核心文件/模块时同步更新工程索引 |

## 四、数据分析前置规范

### 4.1 数据驱动决策强制规则（宪法§2.3、§2.4）

```
规则 D.1: 所有交易决策前必须调取数据分析部门报告
规则 D.2: 趋势报告 (trend_chart.html) 必须存在且时效 < 24h
规则 D.3: 复盘必须引用趋势报告，对比"决策时" vs "当前"
规则 D.4: 审计报告必须包含 data_analysis_ref 字段
规则 D.5: 无趋势报告的复盘 = 无效复盘，必须重做
```

### 4.2 数据分析报告检查清单

- [ ] 日线趋势判断（UP/DOWN/SIDEWAYS）
- [ ] 4H时机分析
- [ ] 阻力分解图
- [ ] E[R]映射图
- [ ] 趋势连续性图表
- [ ] 文件时效 < 24h

## 五、任务创建规范

### 5.1 任务创建集成要求（宪法§13.1）

| 部门 | 是否需要集成 dream-task-creator | 任务类型 |
|------|-------------------------------|----------|
| A6 情报监控部 | ✅ 已集成 | 定时监控、P0响应 |
| auto-repair 自动修复部 | ✅ 已集成 | 修复任务、健康检查 |
| boss-secretary 秘书部 | ✅ 已集成 | 定时汇总、邮件投递 |
| dream-production-consultation-actuator | ✅ 已集成 | 执行任务、落地任务 |
| A4 战术验证部 | ✅ 已集成 | 验证任务、试探任务 |
| A7 盘后审计部 | ✅ 已集成 | 审计任务、复盘任务 |

### 5.2 任务创建流程

```
Step 1: 判断是否需要创建任务
Step 2: 加载 dream-task-creator SKILL
Step 3: 创建任务目录 (mkdir -p ~/.workbuddy/automations/<task-name>/)
Step 4: 写入 automation.toml 配置文件
Step 5: 验证创建结果（文件存在 + 内容正确）
Step 6: 报告完成（输出摘要 + 写入日志）
```

## 六、违规检测与处罚规范

### 6.1 违规类型定义（dream-governance-manager）

| 代码 | 名称 | 严重度 | 处罚 |
|------|------|--------|------|
| V001 | 跳步违规（未查FAQ直接分析） | 🔴 P1 | 记过 + 补齐FAQ |
| V002 | FAQ未查阅 | 🔴 P1 | 警告 + 暂停任务 |
| V003 | FAQ未更新 | 🟠 P2 | 记录 + 补充FAQ |
| V004 | 归档缺失 | 🟡 P3 | 记录 + 补齐归档 |
| V005 | 重复违规（同问题踩2次） | 🔴 P1 | 强制审计 |
| V006 | 伪造记录 | 🔴 P0 | 全系统紧急审计 |

### 6.2 处罚执行流程

```
违规判定 → 查找累计 → 确定等级 → 执行处罚 → 记录 → 通知 → 整改跟踪
```
